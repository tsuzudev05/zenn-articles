title: "依存の方向をコードで検証する ─ レイヤーアーキテクチャの守り方"
---

## はじめに

「Domain層はフレームワークを知らない」と設計原則に書いても、それを人間の注意力だけで守り続けるのは難しい。

実際、ホシカの開発でもドメイン層の `RepositoryError` に `#[from] sqlx::Error` が混入していた。構文エラーは出ない。テストも通る。しかし **Domain層がSQLxに依存している**という、Clean Architectureの根幹を破る違反だった。

このような違反を発見するのに使えるのが、**レイヤー依存チェックスクリプト**だ。


## なぜ違反は気づきにくいのか

```rust
// domain/repositories/wish_item_repository.rs
#[derive(Debug, thiserror::Error)]
pub enum RepositoryError {
    #[error("database error: {0}")]
    Database(#[from] sqlx::Error),  // ← Domain層なのにSQLxを参照している
    #[error("not found")]
    NotFound,
}
```

`#[from] sqlx::Error` は「sqlxのエラーをRepositoryErrorに自動変換する」という便利なマクロだ。infrastructure層で `?` 演算子を使うために書かれたと思われる。

しかし、このコードはdomain crateに置かれている。`use sqlx` という明示的なimportがなくても、`sqlx::Error` という型を参照した時点でDomain層はSQLxに依存する。コンパイルエラーにならないから気づきにくい。

```
依存の方向（理想）:
  Infrastructure → Domain

依存の方向（実態）:
  Infrastructure → Domain → sqlx  ← Domain が外部クレートを知っている
```

Clean Architectureの根本にある「依存逆転の原則」が崩れている。


## 修正方針

Domain層から `sqlx::Error` への依存を取り除き、代わりに汎用的なエラー変数を置く。

```rust
// domain/repositories/wish_item_repository.rs（修正後）
#[derive(Debug, thiserror::Error)]
pub enum RepositoryError {
    #[error("not found")]
    NotFound,
    #[error("unexpected error: {0}")]
    Unexpected(String),  // SQLxを知らない。文字列でラップするだけ
}
```

`sqlx::Error` から `RepositoryError` への変換はinfrastructure層の責務に移す。

```rust
// infrastructure/db/postgres_wish_item_repository.rs
fn to_repo_err(e: sqlx::Error) -> RepositoryError {
    RepositoryError::Unexpected(e.to_string())
}
```

これで依存の方向が正しくなった。

```
修正後:
  Infrastructure（sqlx::Errorをラップして変換）→ Domain（RepositoryError）
  Domain は sqlx を知らない
```


## 依存違反を検出するスクリプト

この違反を毎回手動で探すのは非現実的だ。grepで検出するスクリプトを用意した。

```bash
#!/usr/bin/env bash
# scripts/check-layer-deps.sh

set -euo pipefail

SRC_DIR="$(cd "$(dirname "$0")/../src" && pwd)"
violations=0

check() {
    local layer="$1"      # domain / application
    local forbidden="$2"  # 禁止するキーワード（正規表現）
    local label="$3"

    local matches
    matches=$(grep -rn --include="*.rs" -E "$forbidden" "$SRC_DIR/$layer" 2>/dev/null || true)

    if [ -n "$matches" ]; then
        echo "FAIL  $layer/ が $label に依存しています"
        echo "$matches" | sed 's/^/       /'
        violations=$((violations + 1))
    else
        echo "OK    $layer/ は $label に依存していません"
    fi
}

check "domain"      "axum|sqlx"  "axum / sqlx"
check "application" "axum"       "axum"

[ "$violations" -eq 0 ] && exit 0 || exit 1
```

実行すると依存違反があれば即座に分かる。

```
=== Layer dependency check ===

OK    domain/ は axum / sqlx に依存していません
OK    application/ は axum に依存していません

All checks passed.
```

違反がある場合はファイルパスと行番号を出力してexit 1で終わる。


## CIに組み込む

このスクリプトをGitHub Actionsに追加すれば、プルリクエストのたびに自動検出できる。

```yaml
# .github/workflows/rust.yml に追加
- name: Check layer dependencies
  run: bash scripts/check-layer-deps.sh
```

コードレビューの「依存の方向が正しいか確認する」という作業が自動化される。


## この違反が教えてくれたこと

`#[from] sqlx::Error` を書いた動作としての意図は「infrastructure層でエラーハンドリングを楽にする」だった。動機は正しい。しかし**置く場所を間違えた**。

| 項目 | 修正前 | 修正後 |
|---|---|---|
| `RepositoryError` の定義場所 | domain層 | domain層（変わらず） |
| `From<sqlx::Error>` の実装 | domain層（マクロ自動生成） | infrastructure層（手動） |
| domain の sqlx 依存 | あり | なし |

「便利に書く」ことと「正しい場所に書く」ことはトレードオフになることがある。便利さのためにレイヤー境界を越えるコードは、設計の腐敗の起点になりやすい。

自動テストが「動作の正しさ」を守るように、このスクリプトは「設計の正しさ」を守る。両方を組み合わせることで、変更に耐えるコードベースを維持できる。
