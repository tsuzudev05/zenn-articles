## はじめに

ドメイン層・アプリケーション層の実装が揃った。しかしテストを走らせようとすると、障壁がある。

`PostgresWishItemRepository` は `PgPool`（PostgreSQLへの接続プール）を必要とする。つまり `AddWishItemUseCase` をテストするためだけに、Dockerを起動してDBを用意してマイグレーションを実行する必要がある。

```
cargo test を実行するには…
  PostgreSQLが起動していること
  DATABASE_URL が設定されていること
  マイグレーションが完了していること
```

これは重すぎる。ドメインロジックを1行変えるたびにDockerを立ち上げるのは、フィードバックループが遅すぎてDDDの実践にならない。

解決策が **InMemoryRepository** だ。

---

## InMemoryRepositoryとは何か

データをデータベースではなく、プログラムのメモリ（Hashマップ）上に保持する、テスト専用のリポジトリ実装だ。

ポイントは「**実装が差し替わるだけで、ユースケースのコードは1行も変わらない**」ことだ。

```rust
// 本番用
let wish_item_repo: Arc<dyn WishItemRepository> =
    Arc::new(PostgresWishItemRepository::new(pool));

// テスト用
let wish_item_repo: Arc<dyn WishItemRepository> =
    Arc::new(InMemoryWishItemRepository::new());
```

`AddWishItemUseCase` はどちらの実装が渡されても気にしない。`WishItemRepository` という trait に依存しているだけで、PostgreSQLもHashMapも知らない。これが**依存逆転の原則**の実体だ。

## 共通ストアを先に作る

3つのリポジトリをそのまま実装すると、全てに同じパターンが現れる。

```rust
// 3つのファイルで全く同じ構造が繰り返される
store: Arc<Mutex<HashMap<Uuid, WishItem>>>,
store: Arc<Mutex<HashMap<Uuid, Budget>>>,
store: Arc<Mutex<HashMap<Uuid, Category>>>,
```

`Arc<Mutex<HashMap<Uuid, T>>>` の操作（取得・全件取得・保存・削除）は共通だ。これを `InMemoryStore<T>` というジェネリック構造体に抽出することで、重複を排除できる。

```rust
/// UUID をキーとするスレッドセーフなインメモリストア。
/// 各インメモリリポジトリが共通で必要とする CRUD 操作を提供する。
pub struct InMemoryStore<T> {
    data: Arc<Mutex<HashMap<Uuid, T>>>,
}

impl<T: Clone + Send + 'static> InMemoryStore<T> {
    pub fn new() -> Self { ... }

    /// 初期データを持つストアを生成する
    pub fn with_items(items: impl IntoIterator<Item = (Uuid, T)>) -> Self { ... }

    pub async fn find_by_id(&self, id: Uuid) -> Option<T> { ... }
    pub async fn find_all(&self) -> Vec<T> { ... }
    pub async fn save(&self, id: Uuid, item: T) { ... }

    /// 削除成功なら true、ID が存在しなければ false を返す
    pub async fn remove(&self, id: Uuid) -> bool { ... }
}
```

`Mutex` を使う理由は、各リポジトリ trait が `Send + Sync` を要求するからだ。複数の非同期タスクから同時にアクセスされる可能性があるため、スレッドセーフであることが前提になっている。

## 3つのInMemoryRepositoryを実装する

`InMemoryStore<T>` を使うと、各リポジトリは薄いラッパーになる。

### InMemoryWishItemRepository

```rust
pub struct InMemoryWishItemRepository {
    store: InMemoryStore<WishItem>,
}

impl InMemoryWishItemRepository {
    pub fn new() -> Self {
        Self { store: InMemoryStore::new() }
    }
}

#[async_trait]
impl WishItemRepository for InMemoryWishItemRepository {
    async fn find_by_id(&self, id: Uuid) -> Result<Option<WishItem>, RepositoryError> {
        Ok(self.store.find_by_id(id).await)
    }

    async fn find_all(&self) -> Result<Vec<WishItem>, RepositoryError> {
        Ok(self.store.find_all().await)
    }

    async fn save(&self, item: &WishItem) -> Result<(), RepositoryError> {
        self.store.save(item.id(), item.clone()).await;
        Ok(())
    }

    async fn delete(&self, id: Uuid) -> Result<(), RepositoryError> {
        if !self.store.remove(id).await {
            return Err(RepositoryError::NotFound);
        }
        Ok(())
    }
}
```

`save()` は同じIDで呼ぶと上書きになる（upsert）。`WishItem` がステータス遷移した後に `save()` を呼ぶと、更新後の状態が正しく保存される。

`delete()` は存在しないIDに対して `RepositoryError::NotFound` を返す。`HashMap::remove()` は削除できれば `Some(T)`、なければ `None` を返すため、その結果を使ってエラーを判定している。

```rust
if !self.store.remove(id).await {      // remove が false = 存在しなかった
    return Err(RepositoryError::NotFound);
}
```

### InMemoryBudgetRepository

`find_by_year_month` だけ少し工夫が必要だ。HashMapはIDをキーにしているため、`year_month` での検索は全件取得してから絞り込む。

```rust
async fn find_by_year_month(&self, ym: YearMonth) -> Result<Option<Budget>, RepositoryError> {
    let all = self.store.find_all().await;
    Ok(all.into_iter().find(|b| b.year_month() == ym))
}
```

インメモリのテスト用途ではデータ件数が少ないので全件スキャンで問題ない。本番のPostgreSQL実装では `WHERE year = $1 AND month = $2` のインデックス検索になる。同じ振る舞いを、異なる効率で実現する。これもリポジトリ抽象化の恩恵だ。

### InMemoryCategoryRepository

カテゴリはアプリケーション上「マスターデータ」的な存在で、テスト時には事前に用意しておく必要がある。`InMemoryStore::with_items()` を使うと `with_categories()` が簡潔に書ける。

```rust
pub fn with_categories(categories: Vec<Category>) -> Self {
    Self {
        store: InMemoryStore::with_items(
            categories.into_iter().map(|c| (c.id, c))
        ),
    }
}
```

`Budget` にも同じパターンで `with_budgets()` を用意した。テストのセットアップが一行で書けるようになる。

```rust
let repo = InMemoryCategoryRepository::with_categories(vec![category]);
let repo = InMemoryBudgetRepository::with_budgets(vec![budget]);
```

## 実際にユースケースをテストできるか確認する

InMemoryRepositoryを組み合わせれば、DBなしでユースケースのフルフローをテストできる。

```rust
#[tokio::test]
async fn add_wish_item_returns_output() {
    let category = Category { id: Uuid::new_v4(), name: "書籍".to_string() };
    let category_id = category.id;

    let wish_item_repo = Arc::new(InMemoryWishItemRepository::new());
    let category_repo = Arc::new(
        InMemoryCategoryRepository::with_categories(vec![category])
    );

    let use_case = AddWishItemUseCase::new(wish_item_repo.clone(), category_repo);

    let input = AddWishItemInput {
        name: "Rustプログラミング入門".to_string(),
        price: 3000,
        category_id,
        memo: None,
    };

    let output = use_case.execute(input).await.unwrap();

    assert_eq!(output.name, "Rustプログラミング入門");
    assert_eq!(output.price, 3000);
    assert_eq!(output.status, "Inbox");
}
```

これが `cargo test` だけで動く。DockerもDBも不要だ。

各リポジトリには、より網羅的なテストも用意している。

```rust
// 削除したアイテムが他に影響しないことを確認
#[tokio::test]
async fn delete_does_not_affect_other_items() {
    let repo = InMemoryWishItemRepository::new();
    let item1 = make_item();
    let item2 = make_item();
    repo.save(&item1).await.unwrap();
    repo.save(&item2).await.unwrap();

    repo.delete(item1.id()).await.unwrap();

    assert!(repo.find_by_id(item2.id()).await.unwrap().is_some());
    assert_eq!(repo.find_all().await.unwrap().len(), 1);
}

// 複数月の中から正しい月の予算を取得できることを確認
#[tokio::test]
async fn find_by_year_month_returns_correct_one_among_multiple() {
    let repo = InMemoryBudgetRepository::new();
    repo.save(&make_budget(2026, 5)).await.unwrap();
    repo.save(&make_budget(2026, 6)).await.unwrap();
    repo.save(&make_budget(2026, 7)).await.unwrap();

    let ym = YearMonth::new(2026, 6).unwrap();
    let found = repo.find_by_year_month(ym).await.unwrap().unwrap();
    assert_eq!(found.year_month(), ym);
}
```

## なぜ「証明」と言えるのか

インメモリ実装でテストが通ることには、動作確認以上の意味がある。

**「ドメイン層とアプリケーション層がDBを知らない」ことの証明**になる。

もし `AddWishItemUseCase` のどこかに `sqlx::query!` が混入していれば、このテストはコンパイルすら通らない。インメモリ実装で動くということは、ユースケースが純粋に「ドメインオブジェクトを操作するロジック」だけで書かれていることを意味する。

```
テストが通る = 依存逆転の原則が守られている
テストが通らない = レイヤー境界が漏れている
```

第8章で書いた「依存チェックスクリプト」は静的解析による検証だった。InMemoryRepositoryのテストは**動的な検証**だ。両方を持つことで、設計の正しさを二重に確認できる。

## モジュール配置

InMemoryRepositoryは `src/infrastructure/in_memory/` に置いた。

```
src/infrastructure/
├── db/
│   ├── postgres_wish_item_repository.rs   ← 本番用（sqlx実装）
│   ├── postgres_budget_repository.rs
│   └── postgres_category_repository.rs
└── in_memory/
    ├── in_memory_store.rs                 ← 共通ストア（汎用 CRUD）
    ├── in_memory_wish_item_repository.rs  ← テスト用（InMemoryStore に委譲）
    ├── in_memory_budget_repository.rs
    └── in_memory_category_repository.rs
```

「テスト用だから `#[cfg(test)]` で隠すべきでは？」と思うかもしれない。しかしインメモリ実装はPhase 03で統合テストやE2Eテストを書く際にも使いたい。`infrastructure` 層の中に本番実装と並べて置くことで、「リポジトリのもう一つの実装」として明示的に扱えるようにした。

## この実装から見えてきたこと

InMemoryRepositoryを実装していて気づいたのは、**実装が驚くほど単純だ**ということだ。

PostgreSQL実装は「SQLを書く」「型を変換する」「エラーをハンドルする」という複雑な作業が伴う。一方インメモリ実装は `InMemoryStore` への委譲だけで済む。

さらに、3つのリポジトリを実装する中でコードの重複に気づいた。`Arc<Mutex<HashMap<Uuid, T>>>` の操作が全てのリポジトリで繰り返されていた。これを `InMemoryStore<T>` に抽出したことで、各リポジトリはドメインの都合（`find_by_year_month` のような検索ロジック）だけに集中できるようになった。

この非対称性と整理の過程が Clean Architecture の設計意図を改めて実感させた。ドメインとアプリケーション層は「何をするか」だけを記述し、「どう永続化するか」という詳細はインフラ層に追い出されている。だから実装の差し替えが、呼び出し側に影響を与えずに行える。

Phase 03 でPostgreSQL実装を完成させたとき、ユースケースのテストは引き続きインメモリで高速に動き続ける。統合テストだけがDBを必要とする。この分離がフィードバックループの速さを維持する。
