## はじめに

ドメインモデルの実装を進めていると、こんなコードが生まれていた。

```rust
pub struct Budget {
    balance: i64,  // ← これは何？
}

pub struct WishItem {
    name: String,  // ← どんな文字列？
}
```

`i64` も `String` も間違いではない。しかしこれらは「残高」でも「欲しいものの名前」でもない。**ただの数値と文字列だ。**

型を見ても「何を表しているか」がわからない。型を見ても「どんな制約があるか」がわからない。これを **Primitive Obsession（プリミティブ型への執着）** と呼ぶ。

## 問題1: i64 で表された残高

### 何が問題か

```rust
// BudgetService にあったコード
pub fn will_exceed(budget: &Budget, price: &Price) -> bool {
    budget.balance() - (price.value() as i64) < 0
}
```

このコードを読んで、すぐに意味が取れるだろうか？

- `< 0` の `0` は何を意味する？
- なぜ `i64` と `u64` が混在しているのか？
- 「予算超過」というドメイン概念がどこにも見えない

ドメインエキスパートは「残高が不足するかどうか」を判断したいのに、コードは「整数の差が負かどうか」という計算を見せている。


### Balance 値オブジェクトで解決する

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct Balance(i64);

impl Balance {
    pub fn from_price(price: &Price) -> Self {
        Self(price.value() as i64)
    }

    /// 残高が負（予算超過）かどうか
    pub fn is_exceeded(&self) -> bool {
        self.0 < 0
    }

    /// 指定金額を支払える残高があるか
    pub fn is_sufficient_for(&self, price: &Price) -> bool {
        self.0 >= price.value() as i64
    }

    /// 指定金額を差し引いた新しい残高を返す
    pub fn deduct(&self, price: &Price) -> Self {
        Self(self.0 - price.value() as i64)
    }
}
```

`Budget` が `Balance` を使うと、コードがこう変わる。

```rust
// Budget::would_exceed
pub fn would_exceed(&self, price: &Price) -> bool {
    !self.balance.is_sufficient_for(price)
}

// Budget::record_purchase
pub fn record_purchase(&mut self, actual_price: Price) -> Vec<DomainEvent> {
    let prev_balance = self.balance;
    self.balance = self.balance.deduct(&actual_price);

    if !prev_balance.is_exceeded() && self.balance.is_exceeded() {
        // 超過イベントを発行
    }
}
```

`prev_balance >= 0 && self.balance < 0` という計算の羅列が、`!prev_balance.is_exceeded() && self.balance.is_exceeded()` というドメイン言語に変わった。

コードを読んで何が起きているかが、計算を追わなくても分かる。


## 問題2: String で表された名前

### 何が問題か

元の `WishItem::new()` はこう書かれていた。

```rust
pub fn new(
    name: impl Into<String>,
    // ...
) -> Result<(Self, Vec<DomainEvent>), WishItemError> {
    let name = name.into();
    if name.is_empty() {
        return Err(WishItemError::EmptyName);
    }
    // ...
}
```

`WishItem` が名前のバリデーション責務を持っている。これは問題だ。

`WishItem::new()` が失敗する原因として「名前が空」が混入している。しかし `WishItem` のドメインエラーは本来「ステータス遷移が不正」であるべきで、「名前の形式が不正」ではない。エラーの責務が混在している。


### WishItemName 値オブジェクトで解決する

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct WishItemName(String);

impl WishItemName {
    pub fn new(value: impl Into<String>) -> Result<Self, WishItemNameError> {
        let value = value.into();
        if value.is_empty() {
            return Err(WishItemNameError::Empty);
        }
        Ok(Self(value))
    }

    pub fn value(&self) -> &str {
        &self.0
    }
}
```

`WishItemName` が存在することで、「`WishItemName` 型の値は必ず空文字でない」という保証が型レベルで成立する。

すると `WishItem::new()` はこう変わる。

```rust
// 変更前: Result を返す（名前バリデーション失敗の可能性）
pub fn new(name: impl Into<String>, ...) -> Result<(Self, Vec<DomainEvent>), WishItemError>

// 変更後: 非 Result（WishItemName が渡された時点でバリデーション済み）
pub fn new(name: WishItemName, ...) -> (Self, Vec<DomainEvent>)
```

**`WishItem::new()` が `Result` を返す必要がなくなった。** `WishItemName` を受け取った時点で、それが有効な名前であることは保証されている。バリデーションは入力の境界（ユースケース層）で行う。

```rust
// application/use_cases/add_wish_item.rs
let name = WishItemName::new(input.name)
    .map_err(|e| UseCaseError::DomainError(e.to_string()))?;

let (item, _events) = WishItem::new(name, price, category, memo); // unwrap 不要
```

バリデーションの失敗は「外側（ユースケース）」で処理し、「内側（ドメインモデル）」には有効な値しか渡らない構造になる。


## 値オブジェクトが持つべきもの

`Balance` と `WishItemName` を設計してみて、値オブジェクトのパターンが見えた。

| 責務 | 内容 |
|---|---|
| **バリデーション** | `new()` で不正値を弾く。型を持つことで「この型の値は有効」を保証する |
| **ドメイン操作** | `is_exceeded()`, `deduct()` など、その概念に関係する操作を持つ |
| **不変性** | 値オブジェクトは変更されない。`deduct()` は新しい `Balance` を返す（自身を変えない） |
| **等値性** | 値が同じなら同じもの（エンティティの「ID同一性」とは逆）|


## プリミティブ型を残してよい場合

全ての `i64` と `String` を値オブジェクトに包む必要はない。判断基準はこうだ。

**値オブジェクトにするべき場合**
- その型に固有のバリデーションルールがある（空文字禁止、負値禁止など）
- その型に固有の操作がある（`is_exceeded()`, `deduct()` など）
- プリミティブ型では「何を表すか」が名前から読み取れない

**プリミティブ型のままでよい場合**
- DTOやAPI レスポンスなど、ドメインの外側のデータ（`BudgetStatusOutput { balance: i64 }`）
- ユニークな操作もバリデーションも持たない単純な識別子

ドメイン層の内側でプリミティブ型が多いほど、ビジネスルールはコメントやドキュメントに追い出される。コードが嘘をつく余地が増える。**型をドキュメントとして使う**という意識が、DDDの実践につながる。
