---
title: "エンティティと値オブジェクトの違いをPriceで理解する"
---

## はじめに

DDDを学び始めたとき、エンティティと値オブジェクトの違いが最初はよくわからなかった。

「IDを持つかどうか」という説明を読んでも、「じゃあなぜPriceにIDが必要ないのか」が腑に落ちなかった。

個人開発アプリ「ホシカ」（欲しいものリスト × 予算管理）のドメインモデリングを通じて、この違いが実感として掴めたのでまとめる。


## 結論から言う

判断基準はひとつだ。

**「それをIDで追跡する必要があるか？」**

- 必要あり → **エンティティ**
- 必要なし → **値オブジェクト**


## Priceが値オブジェクトである理由

`Price(1000)` が2つあったとき、それらは**同じもの**だ。どちらの1000円かを区別する意味がない。

```rust
let a = Price(1000);
let b = Price(1000);

// a と b は「同じ」。IDで区別する必要がない。
assert_eq!(a, b); // true
```

もし `Price` にIDを持たせると、こういう奇妙なことが起きる。

```rust
// IDが違えば「別物」になってしまう
let a = Price { id: 1, amount: 1000 };
let b = Price { id: 2, amount: 1000 };

a == b // false ← 1000円と1000円が違うものになる。おかしい。
```

**値が同じなら同じもの**、それが値オブジェクトの本質だ。


## WishItemがエンティティである理由

一方、欲しいものリストのアイテム `WishItem` はどうか。

```rust
let a = WishItem { id: 1, name: "ヘッドホン", price: Price(30000), status: Inbox };
let b = WishItem { id: 1, name: "ヘッドホン", price: Price(50000), status: NextToBuy };
```

`a` と `b` はステータスも価格も違う。しかし `id` が同じなら**同じアイテム**だ。「このヘッドホン」はどれだけ状態が変わっても同一の存在として追跡したい。

これがエンティティの本質だ。**状態が変わっても、IDが同じなら同一のもの**。


## 表で整理する

| | エンティティ | 値オブジェクト |
|---|---|---|
| 同一性の基準 | ID | 値そのもの |
| 状態変化 | 変わっても同一のもの | 変わったら別のもの |
| ミュータビリティ | 変化する（ステータス遷移など） | イミュータブルが原則 |
| ホシカの例 | `WishItem`, `Budget`, `PurchaseRecord` | `Price`, `Category`, `WishItemStatus` |


## Rustで書くとより明確になる

値オブジェクトはイミュータブルで、値で比較する。Rustでは `PartialEq` を derive するだけで表現できる。

```rust
#[derive(Debug, Clone, PartialEq)]
struct Price(u64); // u64 で負の値を型レベルで排除

impl Price {
    pub fn new(amount: u64) -> Self {
        Self(amount)
    }
}
```

エンティティはIDを持ち、IDで同一性を判断する。

```rust
#[derive(Debug)]
struct WishItem {
    id: WishItemId,          // ← 同一性の根拠
    name: String,
    price: Price,            // ← 値オブジェクトを保持
    status: WishItemStatus,  // ← 値オブジェクトを保持
}

impl PartialEq for WishItem {
    fn eq(&self, other: &Self) -> bool {
        self.id == other.id  // IDだけで比較。nameやpriceは関係ない
    }
}
```


## 「価格を変更する」はどう表現するか

値オブジェクトはイミュータブルなので、`Price` 自身は変化しない。`WishItem` の `price` フィールドを**新しい `Price` で差し替える**。

```rust
impl WishItem {
    pub fn update_price(&mut self, new_price: Price) {
        self.price = new_price; // Price(30000) を Price(50000) に差し替え
    }
}
```

「元のPrice」が変化したのではなく、「WishItemが参照するPriceが変わった」という表現になる。


## 値オブジェクトにするメリット

### 1. 不正な値を型レベルで弾ける

`Price` を `u64` でラップしているので、負の価格はコンパイル時に存在できない。`f64` や `i64` を生で使うと `-1000` が入り込む余地があるが、型で防げる。

```rust
// これはコンパイルエラー
let p = Price(-1000); // u64 に負の値は入らない
```

### 2. ドメインルールをオブジェクトに閉じ込められる

「Priceは0円より大きくなければならない」というルールを `Price::new()` に閉じ込めることができる。

```rust
impl Price {
    pub fn new(amount: u64) -> Result<Self, DomainError> {
        if amount == 0 {
            return Err(DomainError::InvalidPrice("0円は無効です".into()));
        }
        Ok(Self(amount))
    }
}
```

### 3. テストが書きやすい

`Price(3000) == Price(3000)` が当然に成立するため、テストの assertion がシンプルになる。


## 「これはどちらか？」で迷ったとき

ホシカの設計でも迷ったものがある。

### PurchaseRecordはなぜエンティティか

購入記録 `PurchaseRecord` は値オブジェクトにも見えるが、エンティティにした。

理由は、**同じ日に同じ金額で同じアイテムを2回購入した場合を区別したい**から。`PurchaseRecord(2026-06-01, 3000円, ヘッドホン)` が2件あったとき、それらは別の購入という事実であり、IDで追跡する必要がある。

### WishItemStatusはなぜ値オブジェクトか

ステータス `WishItemStatus` は `enum` で定義している。`Inbox` は `Inbox` だ。「このInboxとあのInboxは別物」という区別は意味をなさない。値で比較できるので値オブジェクト。


## まとめ

エンティティと値オブジェクトを分ける問いはシンプルだ。

> **「それをIDで追いかける必要があるか？」**

`Price` は「1000円」という値そのものが同一性を持つ。追いかける必要がない → 値オブジェクト。

`WishItem` は「このヘッドホン」という存在を状態が変わっても追跡したい → エンティティ。

Rustで書くと、この違いは構造体の設計に自然に現れる。値オブジェクトは `#[derive(PartialEq)]` で値比較、エンティティは `id` フィールドを持ち `PartialEq` をIDで実装する。型システムがドメインモデルを表現する手段になるのがDDDとRustの相性の良さだと感じている。
