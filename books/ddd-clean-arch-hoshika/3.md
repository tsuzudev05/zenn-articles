## はじめに

DDDを学ぶとき、最初に「ユビキタス言語を定義せよ」と言われる。しかし「どうやって決めるのか」「何を書けばいいのか」が最初はわからなかった。

ホシカ（欲しいものリスト × 予算管理アプリ）の用語集を実際に設計した過程をまとめる。


## ユビキタス言語とは何か

**コード・会話・ドキュメントすべてで使う、統一された言葉の辞書。**

「ユビキタス（ubiquitous）」は「どこにでも存在する」という意味だ。会議でも、コードでも、仕様書でも、同じ言葉を使う。

なぜ必要か。言葉がズレると認識がズレ、認識のズレがバグになるからだ。

```
仕様書: 「欲しいもの」
会話:   「アイテム」「商品」
コード: item, product, goods, target
```

これが同じ概念を指しているとしたら、コードを読むたびに変換コストが発生する。長期的には設計の歪みにもなる。

ユビキタス言語を決めると、コードが仕様書になる。


## どうやって決めるか

ホシカでの手順を振り返ると、こういう流れだった。

### 1. 「このアプリで何が起きているか」を動詞で書き出す

- 欲しいものを登録する
- レビューして「次に買う」「保留」「不要」を決める
- 購入する
- 予算を確認する

### 2. 名詞を抽出して整理する

- 欲しいもの → `WishItem`
- 予算 → `Budget`
- 購入記録 → `PurchaseRecord`
- レビューの結果（次に買う/保留/不要）→ `ReviewDecision`

### 3. 迷ったらビジネスの文脈で考える

`WishItem` か `DesiredItem` か迷った。英語として自然なのは `DesiredItem` だが、「欲しいものリスト」というアプリの文脈では `WishList` の `WishItem` が最もしっくりくる。技術的な英語の正確さより**ドメインの言葉**を優先した。


## ホシカで定義した用語集

### エンティティ（IDで同一性を追跡するもの）

| 用語 | 日本語 | 定義 |
|------|--------|------|
| `WishItem` | 欲しいもの | 欲しいものリストの1件。名前・価格・カテゴリ・登録日・ステータスを持つ。IDで同一性を追跡する |
| `Budget` | 予算 | 月次の使用可能金額。月・設定金額・残高を持つ。同一月に1件のみ存在する |
| `PurchaseRecord` | 購入記録 | WishItemを購入した際に生成される。購入日・実支払額・WishItemへの参照・メモを持つ |

`PurchaseRecord` を独立したエンティティにした理由がある。希望価格と実支払額は異なることがある（セール、値下がり）。この差を記録しないと家計の実態が歪む。WishItemに実支払額を持たせると責務が混濁するため、購入の文脈は `PurchaseRecord` に分離した。

### 値オブジェクト（値で同一性を比較するもの）

| 用語 | 日本語 | 定義 |
|------|--------|------|
| `Price` | 金額 | 0円以上の正の値。`u64` で型レベルに負の値を排除する |
| `Category` | カテゴリ | アイテムの分類（書籍・ガジェット・ファッション等）。文字列ではなく型として扱う |
| `WishItemStatus` | ステータス | `Inbox` / `NextToBuy` / `OnHold` / `Archived` / `Purchased` のいずれか |
| `Memo` | メモ | 購入記録に添付できる自由記述。空可。最大500文字 |
| `YearMonth` | 年月 | 予算の対象月。月は1〜12のみ有効 |

### ドメインサービス（複数集約をまたぐロジック）

| 用語 | 日本語 | 定義 |
|------|--------|------|
| `BudgetService` | 予算サービス | 購入時に予算残高を確認し、超過する場合は `BudgetExceeded` イベントを発生させる。`WishItem` と `Budget` をまたぐためドメインサービスとして定義 |

### ドメインイベント（何が起きたかの記録）

| イベント名 | 発生条件 |
|------------|----------|
| `ItemAdded` | WishItemがInboxに追加された |
| `ItemReviewed` | ユーザーがレビューしてステータスを変更した |
| `ItemMovedToNextToBuy` | レビューでNextToBuyに変更された（購入候補になった） |
| `ItemArchived` | 「不要」と判断されアーカイブされた |
| `ItemPurchased` | 購入され、PurchaseRecordが生成された |
| `BudgetSet` | 月次予算が設定された |
| `PurchaseRecorded` | PurchaseRecordがBudgetに記録された |
| `BudgetExceeded` | 購入によって残高が0を下回った |


## 言語を決めるとコードが変わる

用語集が定まると、コードの命名に迷わなくなる。

```rust
// 用語集がない場合
fn add_item(name: String, price: f64) -> Item { ... }
fn check(item_id: i64) -> bool { ... }
fn buy(id: i64) { ... }

// ユビキタス言語に基づく場合
fn add_wish_item(name: String, price: Price) -> WishItem { ... }
fn review_wish_item(id: WishItemId, decision: ReviewDecision) -> WishItem { ... }
fn purchase(id: WishItemId, actual_price: Price) -> PurchaseRecord { ... }
```

後者はコードを読むだけで何をしているかがわかる。コードが仕様書を兼ねる状態になる。


## 用語集を作る過程で起きたこと

### 削除した概念

`WaitingPeriod`（待機期間）という値オブジェクトを最初は設計に含めていた。「登録から7日間は購入できない」という仕組みを想定していた。

しかし衝動買い防止の本質を考え直すと、それはタイマーによる強制ではなく**「リストを見直す習慣」**だと気づいた。`WaitingPeriod` はドメインに存在しない概念だったので削除した。

用語集を整理する過程で「実際のビジネスルールと乖離した概念」を取り除けた。

### `WishItemStatus` を `int` フラグにしなかった理由

```rust
// int フラグの場合
status: i32 // 0=inbox, 1=next_to_buy, 2=on_hold... これは何？

// 値オブジェクトの場合
status: WishItemStatus // Inbox / NextToBuy / OnHold / Archived / Purchased
```

前者は読む人がドキュメントを参照しないと意味がわからない。後者はコードを読めば状態が伝わる。ユビキタス言語の原則を型レベルで表現している。


## まとめ

ユビキタス言語の設計は地味に見えるが、「コードを書く前に最も重要な作業」だと感じた。

- 動詞で「何が起きるか」を書き出す
- 名詞を抽出してエンティティ・値オブジェクトに分類する
- 迷ったらビジネスの文脈で判断する

用語集が固まると、コードの命名に迷わなくなる。設計上の議論も「この概念は用語集に存在するか」という問いで整理できる。
