## はじめに

ホシカ（欲しいものリスト × 予算管理アプリ）のドメインモデルとRustの型設計が固まったあと、次の課題はPostgreSQLへの永続化だ。

DynamoDBやMongoDBを使ってきた自分にとって、RDBのテーブル設計は「ドキュメント設計とどう違うのか」がすぐには掴めなかった。設計を進める中で学んだ考え方の違いと、ドメインルールをDBに落とし込むうえで行った判断を記録する。


## NoSQLとRDBの根本的な違い

NoSQLでは「どうデータを取り出すか」を先に決め、それに最適なドキュメント構造を設計する。DynamoDBなら最初にアクセスパターンを洗い出すのが定石だ。

RDBの設計原則は逆に近い。**データの意味・役割を正確に表現することを優先**し、「DBが嘘をつかない」ことを目標にする。

| | NoSQL（DynamoDB / MongoDB） | PostgreSQL |
|---|---|---|
| データの単位 | ドキュメント（JSON的な構造） | 行と列の表 |
| 関連データ | 1ドキュメントに埋め込むことが多い | 別テーブルに分けて外部キーで繋ぐ |
| 不変条件の担保 | アプリ側で書く | DB制約（UNIQUE / CHECK / FK）で守る |
| 参照整合性 | アプリ側で保証する必要がある | 外部キー制約がDBで自動的に弾く |

アプリ側のバリデーションが漏れても、DB制約が最後の砦になる。これがRDBの強みだ。


## ドメインモデルとテーブルの対応

ホシカのドメインモデルは2つの境界づけられたコンテキストに分かれている。これをそのままテーブル構造に反映した。

```
categories       ← Category（値オブジェクト）
wish_items       ← WishItem（集約ルート）
budgets          ← Budget（集約ルート）
purchase_records ← PurchaseRecord（エンティティ）
```

NoSQLなら `wish_items` コレクションに `categoryName: "書籍"` を埋め込む方法もある。しかしそうすると、カテゴリ名を変更したとき全件更新が必要になる。`categories` テーブルを分離することで、この問題を避けられる。

**「役割が違うデータは別テーブル」** がRDBの基本だ。


## PostgreSQLの型がドメインルールを表現する

### ENUM型でステータス遷移を守る

Rustで定義した `WishItemStatus` をPostgreSQLでも同じ形で表現できる。

```sql
CREATE TYPE wish_item_status AS ENUM (
    'Inbox',
    'NextToBuy',
    'OnHold',
    'Archived',
    'Purchased'
);
```

Rustの `enum WishItemStatus` と対称性を持たせることで、コードとDBを読み比べやすくなる。そして許可外のステータスはDBレベルで弾かれる。アプリのバグで `'Reviewing'` のような存在しないステータスが入り込む余地がない。

なおステータス**遷移**のルール（`Inbox → NextToBuy` のみ可など）はCHECK制約では表現できない。「前の値に依存する条件」はDBのCHECKで書けないからだ。この部分はRustのドメインオブジェクトが担保する。

### 外部キーはドメインの参照整合性そのもの

```sql
category_id UUID NOT NULL REFERENCES categories(id)
```

DynamoDBでは「参照先が存在するか」はアプリ側でチェックするしかなかった。`REFERENCES` を使うと、`categories` に存在しないUUIDを `wish_items.category_id` に入れようとした瞬間にDBがエラーを返す。

アプリのバグでカテゴリが消えた後も `wish_items` が残る、という不整合が**構造的に起きえない**設計になる。

### BIGINTと符号なし整数の違い

PostgreSQLの整数型は全て**符号付き**だ。Rustの `u64` に相当する型は存在しない。

```sql
price BIGINT NOT NULL CHECK (price >= 0)
```

Rustで `Price(u64)` と型レベルで負の値を排除できても、DBでは `BIGINT`（符号付き64bit整数）で格納するため `CHECK (price >= 0)` が必要になる。Rustの型システムとDBの制約は役割分担しながら同じ不変条件を守る。


## memo設計：Option<String>とNULLの扱い

`memo` 列の設計は、最初の案から方針が変わった。

### 最初の案：NOT NULL DEFAULT ''

```sql
memo TEXT NOT NULL DEFAULT ''
```

Rustで `Memo(String)` という型を使い、「メモなし = 空文字」で統一する案だ。

しかしこれは問題がある。**「メモがない」と「空文字のメモ」は意味が違う。** `""` でNULLを代替することはドメイン的に不自然だ。

### 採用した案：NULL許容 + 空文字禁止

```sql
memo TEXT CHECK (memo IS NULL OR memo <> '')
```

Rust側は `Option<String>`。

- `NULL` / `None` → メモなし
- 非空文字列 / `Some("...")` → メモあり

`Some("")` はDBのCHECK制約で弾かれる。これでRust側も「`Some` なら必ず何か書いてある」という前提で実装できる。アプリ層に「`None` と `Some("")` を同一視する」という曖昧な取り決めが不要になる。

| 状態 | DB | Rust |
|------|-----|------|
| メモなし | `NULL` | `None` |
| メモあり | `'買ったら後悔しそう'` | `Some("買ったら後悔しそう")` |
| 空文字（禁止） | CHECK制約でエラー | アプリ層でバリデーション |


## balance設計：あえてCHECK制約をつけない

`budgets` テーブルの `balance`（残高）には意図的に `CHECK (balance >= 0)` をつけていない。

```sql
balance BIGINT NOT NULL
-- CHECK (balance >= 0) はあえてつけない
```

理由はドメインルールだ。「残高が負になる = 予算超過」という状態はホシカで正当に発生する。この状態を `BudgetExceeded` イベントとして通知することがドメインの設計意図であり、DBで「起きえない」ことにしてはいけない。

**DB制約は「ドメイン上ありえない値」を弾くためにある。** 「ありえるが問題がある状態」まで弾いてしまうと、ドメインの表現力が落ちる。


## マイグレーション順序は依存グラフ

RDBでは外部キーの参照先テーブルが先に存在していないとテーブルを作れない。

```
1. categories        ← 他から参照されるので最初
2. wish_items        → categories を参照
3. budgets           ← purchase_records から参照される
4. purchase_records  → budgets と wish_items の両方を参照
```

これはドメインモデルの依存関係と対応している。`WishItem` が `Category` を参照するというドメインの構造が、テーブル作成の順序にそのまま現れる。

DynamoDBにはこの概念がなく、テーブルをどの順序で作っても問題なかった。RDBではテーブル定義ファイルの書き順も設計の一部になる。


## 設計原則のまとめ

ドメインモデルをRDBに落とし込む過程で気づいた原則を整理する。

**1. テーブルの都合でドメインを歪めない**

`NOT NULL DEFAULT ''` でメモをNULL排除するのは「テーブル設計を楽にするためにドメインを歪める」行為だった。`Option<String>` というドメインの意図をDB設計にも反映すべきだった。

**2. DB制約はドメインルールの最後の砦**

Rustの型システム、アプリ層のバリデーション、DB制約の三層でドメインルールを守る。外側が漏れても内側が守る。DB制約がなければ最後の砦がない。

**3. 「ありえない」と「問題がある」を区別する**

`balance < 0` は「問題がある状態」だが「ありえない状態」ではない。CHECK制約で弾くのは前者ではなく後者だけでいい。`price < 0` や `month > 12` は本当に「ありえない」のでCHECKで弾く。

NoSQLで培った「アプリ側で全てを守る」という習慣を見直し、DBに守らせられるものはDBに委ねることがRDBとの正しい付き合い方だと感じている。
