---
title: "衝動買い防止のドメインルールをどう設計したか"
---

## はじめに

ホシカ（欲しいものリスト × 予算管理アプリ）のコアコンセプトは「衝動買い防止」だ。

「衝動買いを防ぐ」というビジネスルールをどうドメインモデルで表現するか。最初の設計案から最終的な答えが出るまでの過程をまとめる。


## 最初の設計案：タイマーで強制する

「衝動買い」というと、こういう仕組みを思いつきやすい。

```
登録 → 待機期間（7日）→ 「今も欲しいですか？」確認 → 購入可
```

これをドメインモデルで表現しようとすると、`WaitingPeriod`（待機期間）という値オブジェクトが必要になる。

```rust
struct WishItem {
    id: WishItemId,
    name: String,
    price: Price,
    waiting_period: WaitingPeriod, // 7日間
    created_at: DateTime,
}

impl WishItem {
    pub fn can_purchase(&self, now: DateTime) -> bool {
        now >= self.created_at + self.waiting_period.duration()
    }
}
```

一見よさそうだが、これは**本当の問題を解決していない**と気づいた。


## なぜタイマーでは不十分か

タイマーで7日待たせても、ユーザーは「早く7日経たないかな」と思いながら待つだけだ。「本当に欲しいか」を考えるきっかけにはならない。

衝動買いを防ぐのは**「立ち止まって考える行為」**であり、システムが自動的に止める仕組みではない。

この気づきがドメインルールの再定義につながった。


## 本質のドメインルール

**衝動買いを防ぐのは「リストを見直す」という行為そのもの。**

```
登録（ItemAdded）
    ↓
WishItemStatus: Inbox（まだ買えない）
    ↓  ← この状態自体が「一度立ち止まる」ことを強制する
ユーザーが能動的にレビューする（ItemReviewed）
    ├─ 次に買う → NextToBuy → 購入可能
    ├─ 保留     → OnHold    → リストに残る
    └─ 不要     → Archived  → 非表示（履歴として保持）
```

登録した瞬間は `Inbox`（未整理）状態で、購入できない。ユーザーが自分でリストを眺めて「次に買う」「保留」「不要」と判断する行為を経て初めて購入候補になる。

タイマーは不要だ。`WaitingPeriod` という概念を削除した。


## WishItemStatusの設計

この設計の核心は `WishItemStatus` という値オブジェクトのステータス遷移だ。

```rust
#[derive(Debug, Clone, PartialEq)]
enum WishItemStatus {
    Inbox,      // 登録直後。購入不可
    NextToBuy,  // レビューで「次に買う」と決定。購入可能
    OnHold,     // レビューで「保留」。引き続きリストに残る
    Archived,   // レビューで「不要」。非表示（終端状態）
    Purchased,  // 購入済み（終端状態）
}
```

遷移ルール（不変条件）は `WishItem` のメソッドに閉じ込める。

```rust
impl WishItem {
    pub fn review(&mut self, decision: ReviewDecision) -> Result<DomainEvent, DomainError> {
        match (&self.status, &decision) {
            (WishItemStatus::Archived, _) => Err(DomainError::CannotReviewArchivedItem),
            (WishItemStatus::Purchased, _) => Err(DomainError::CannotReviewPurchasedItem),
            (_, ReviewDecision::NextToBuy) => {
                self.status = WishItemStatus::NextToBuy;
                Ok(DomainEvent::ItemMovedToNextToBuy { id: self.id })
            }
            (_, ReviewDecision::OnHold) => {
                self.status = WishItemStatus::OnHold;
                Ok(DomainEvent::ItemReviewed { id: self.id, new_status: WishItemStatus::OnHold })
            }
            (_, ReviewDecision::Archive) => {
                self.status = WishItemStatus::Archived;
                Ok(DomainEvent::ItemArchived { id: self.id })
            }
        }
    }

    pub fn purchase(&mut self, actual_price: Price) -> Result<DomainEvent, DomainError> {
        match &self.status {
            WishItemStatus::NextToBuy => {
                self.status = WishItemStatus::Purchased;
                Ok(DomainEvent::ItemPurchased { id: self.id, actual_price })
            }
            _ => Err(DomainError::CannotPurchaseItemNotInNextToBuy),
        }
    }
}
```

**`Archived` と `Purchased` からの遷移は不可（終端状態）。`NextToBuy` 以外から `purchase()` は呼べない。** これらの不変条件はドメインオブジェクト自身が守る。Application層やPresentation層が確認する必要はない。


## なぜ「不要」はアーカイブで削除しないのか

`Archived`（不要）にしたアイテムは削除せず、非表示にするだけにした。

理由は「買わなかった判断の積み重ね」が価値を持つからだ。

「あのとき買わなくて正解だった」という振り返りができる。また、半年後に同じものを「欲しい」と思ったとき、過去に「不要」と判断した記録があれば冷静に考えられる。

ドメインのルールとしては:
- `Archived` は「存在しなかったことにする」のではなく「非表示にする」
- 履歴は消さない

削除と非表示は技術的には似ているが、**ビジネスの意図が違う**。DDDではこの差を型で表現することが大事だ。


## 設計変更のまとめ

| | 最初の案 | 最終的な設計 |
|---|---|---|
| 仕組み | タイマー（7日待機） | レビューという行為 |
| 概念 | `WaitingPeriod` あり | `WaitingPeriod` なし |
| ステータス | 時間経過で自動変化 | ユーザーの判断で変化 |
| 衝動買い防止の主体 | システム | ユーザー |


## この設計から得た気づき

ユビキタス言語を整理する過程で「ドメインに実在しない概念」を除去できた。

`WaitingPeriod` は「衝動買い防止らしい概念」だが、**本当のビジネスルールと乖離していた**。コードを書く前に言語を整理したからこそ気づけた。

DDDの「ドメインエキスパートと対話しながらモデルを育てる」という考え方を実感した瞬間だった。個人開発でも「ユーザーとしての自分」と「開発者としての自分」を分けて対話することで同じプロセスが成立する。
