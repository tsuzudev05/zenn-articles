title: "DDDとClean Architectureはそれぞれ何を解決するのか"

## はじめに

DDDとClean Architectureはよく一緒に語られる。しかし「なぜ2つ必要なのか」「どう違うのか」が最初はわからなかった。

ホシカ（欲しいものリスト × 予算管理アプリ）を設計する中でこの問いに向き合ったので、自分なりの理解をまとめる。


## DDDが解決したいこと

**「コードとビジネスの言語が乖離していく」問題。**

ソフトウェアが複雑になるにつれて、コードはビジネスの実態から離れていく。典型的な症状はこうだ。

- テーブル名・クラス名が技術都合の言葉になる（`t_user_mst`、`DataManager`）
- 仕様書・会話・コードで異なる言葉が使われる
- 「このフラグ何？」「あの処理どこ？」が増える

認識のズレが積み重なり、バグや設計の歪みを生む。

DDDの答えはシンプルだ。**ビジネスの言葉をそのままコードにする。**

```rust
// 技術都合の命名
fn update_flg(id: i64, flg: i32) { ... }

// ユビキタス言語で書くと
fn review_wish_item(id: WishItemId, decision: ReviewDecision) { ... }
```

後者はコードを読むだけで「欲しいものをレビューする」というビジネスルールが伝わる。ドキュメントを開く必要がない。

### DDDの主な概念

| 概念 | 役割 |
|---|---|
| **ユビキタス言語** | チーム全員が使う共通の言葉 |
| **エンティティ** | IDで同一性を追跡するオブジェクト |
| **値オブジェクト** | 値そのものが同一性を表すオブジェクト |
| **集約** | 一貫性を保つオブジェクトのまとまり |
| **ドメインサービス** | 複数集約をまたぐロジックの置き場所 |
| **ドメインイベント** | 「何が起きたか」という事実の記録 |
| **バウンデッドコンテキスト** | 言葉の意味が有効な境界 |

DDDが問いかけるのは**「何をモデリングするか」**だ。


## Clean Architectureが解決したいこと

**「フレームワークやDBへの依存がビジネスロジックを汚染していく」問題。**

フレームワークは便利だが、その都合がビジネスロジックに染み込むと変更が怖くなる。

- DBをPostgreSQLからDynamoDBに変えたら全体が壊れる
- フレームワークのバージョンを上げたらビジネスロジックの書き直しが必要になる
- テストを書こうとしたらDBが必要になる

これらは「ビジネスロジックが外部依存を知っている」から起きる。

Clean Architectureの答えは**「依存の方向を制御する」**こと。

```
Presentation（Axum handlers）
      ↓
Application（Use Cases）
      ↓
Domain（Entities / Repository traits）
      ↑
Infrastructure（SQLx / 外部API）
```

依存は常に内側へ向かう。**Domain層は外側を知らない。** AxumもSQLxもimportしない。

依存逆転の原則がポイントだ。DBを使いたいなら、Domain層でtraitを定義し、Infrastructure層がそれをimplする。

```rust
// Domain層: DBを知らないインターフェース定義
trait WishItemRepository {
    fn find_by_id(&self, id: WishItemId) -> Option<WishItem>;
    fn save(&self, item: WishItem);
}

// Infrastructure層: PostgreSQL実装（Domain層が依存するのではなく、逆にimplする）
struct PostgresWishItemRepository { ... }

impl WishItemRepository for PostgresWishItemRepository {
    fn find_by_id(&self, id: WishItemId) -> Option<WishItem> {
        // SQLxで実装
    }
    fn save(&self, item: WishItem) { ... }
}
```

DBをインメモリ実装に差し替えてもDomain層は一行も変わらない。これが**変更容易性の実体**だ。

Clean Architectureが問いかけるのは**「どう層を分けるか」**だ。


## 2つの補完関係

| | 問いかけ | 解決する問題 |
|---|---|---|
| **DDD** | 何をモデリングするか | ビジネスとコードの乖離 |
| **Clean Architecture** | どう層を分けるか | 外部依存によるロジックの汚染 |

**DDDがモデルを定義し、Clean Architectureがそのモデルの置き場所を守る。**

DDDだけだと、ビジネスルールを正確にモデル化できても、それがAxumのhandlerやSQLxのクエリと同じファイルに書かれてしまうリスクがある。

Clean Architectureだけだと、層の分け方は綺麗でも、中身がビジネスの実態とズレた命名や設計になりうる。

組み合わせることで、**「言葉が正確で、かつ変更に強い」設計**になる。


## ホシカでの具体的な現れ方

### DDDの現れ方

`wishItem.review(ReviewDecision::NextToBuy)` というコードは、ドメインエキスパート（ユーザー）に見せて「欲しいものをレビューして、次に買うと決めた」と伝わる。これがユビキタス言語。

`WishItemStatus` という値オブジェクトが `Inbox / NextToBuy / OnHold / Archived / Purchased` というビジネスの言葉で状態を表す。`int` フラグや `boolean` では表現できない意図がある。

### Clean Architectureの現れ方

```
src/
├── domain/          ← WishItem, Budget, WishItemRepository trait
├── application/     ← AddWishItemUseCase, ReviewWishItemUseCase
├── infrastructure/  ← PostgresWishItemRepository（SQLx実装）
└── presentation/    ← Axum handlers（薄いラッパー）
```

`domain/` は `use axum` も `use sqlx` も書かない。ドメインオブジェクトのテストはDBなしで `cargo test` だけで通る。


## まとめ

- **DDDが答える問い**: このビジネスルールを何と呼ぶか、どのオブジェクトに責任を持たせるか
- **Clean Architectureが答える問い**: このコードはどのレイヤーに置くか、依存の方向は正しいか

どちらか一方では解決できない問題がある。2つを組み合わせることで初めて「ビジネスルールが正確に表現され、かつ変更に強い」設計が実現する。
