title: "new() と reconstitute() ─ 集約の2つのコンストラクタ"

## はじめに

`Budget::new()` の実装を見ていて、こんな疑問が浮かんだ。

```rust
pub fn new(year_month: YearMonth, amount: Price) -> (Self, Vec<DomainEvent>) {
    let id = Uuid::new_v4();
    let balance = Balance::from_price(&amount); // ← なぜ内部で作る？
    let set_at = Utc::now();
    ...
}
```

「UUID・Balance・現在時刻は外から渡した方が柔軟では？」

```rust
// こういう設計の方が良いのでは？
Budget::new(
    id: Uuid::new_v4(),
    year_month,
    amount,
    balance: Balance::from_price(&amount), // 呼び出し側が作る
    set_at: Utc::now(),
)
```

この疑問への答えが、DDDにおける**集約の不変条件**と**2つのコンストラクタパターン**に繋がっている。

---

## `Balance` を外から渡してはいけない理由

「新規予算の初期残高 = 予算金額」はドメインルールだ。

もし `Balance` を引数で受け取る設計にすると、呼び出し側がこのルールを破れてしまう。

```rust
// ❌ 呼び出し側がルールを壊せる
Budget::new(
    year_month,
    amount: Price::new(50000).unwrap(),
    balance: Balance::new(0),  // amount と一致していない
)
```

`amount = 50000` なのに `balance = 0` の `Budget` が生まれてしまう。これは**不正な状態のまま構築されたエンティティ**だ。

DDDにおける集約ルートの責務は「**自分の不変条件を自分で守ること**」だ。`Budget::new()` が `Balance::from_price()` を内部で呼ぶのは「初期残高の設定は Budget が責任を持つ」という設計の表明だ。

```rust
// ✅ 不変条件はアグリゲート内部に閉じている
pub fn new(year_month: YearMonth, amount: Price) -> (Self, Vec<DomainEvent>) {
    let balance = Balance::from_price(&amount); // ← これがドメインルール
    ...
}
```

「`Balance` を外から渡せるようにする = 不変条件を外に漏らす」ということになる。

---

## では UUID と Utc::now() は？

`Balance` と違い、UUID と現在時刻には別の議論がある。

### UUID — 生成は内部でよい

エンティティの「新規作成」では、IDを内部で生成するのがDDDの標準的な書き方だ。

```rust
let id = Uuid::new_v4(); // 集約が自分のIDを生成する
```

呼び出し側が `Uuid::new_v4()` を呼んで渡す設計にしても動作はする。しかし「Budget を作るときのIDはどこから来るか」という責務が外に漏れる。

### Utc::now() — テスタビリティとのトレードオフ

`Utc::now()` は副作用（I/O）を伴う。厳密な設計では外から注入して純粋関数にする考え方がある。

```rust
// 厳密版: 時刻を注入できる
pub fn new(year_month: YearMonth, amount: Price, now: DateTime<Utc>) -> ...
```

「作成日時が正確に何時か」を検証するテストが必要な場合は有効な選択だ。ただしコードが増える割に多くのケースでその検証は必要ない。`Utc::now()` を直接使う実用的な判断は DDD コミュニティでも一般的に受け入れられている。

## `new()` だけでは足りない ─ `reconstitute()` の必要性

ここまでで「`new()` は不変条件を守るため内部で状態を生成する」という設計方針が見えた。

しかしここで問題が生じる。**DBから Budget を復元するとき**だ。

PostgreSQLに保存された Budget には、すでに以下が存在する。

- 過去に生成された UUID（再生成してはいけない）
- `record_purchase()` によって減少した残高（`amount` と一致しない）
- 保存時の `set_at`（現在時刻ではない）

これを `Budget::new()` で復元しようとすると、すべてが上書きされてしまう。**復元は「新規作成」ではない。**

この問題を解決するのが `reconstitute()`（再構築）コンストラクタだ。

```rust
impl Budget {
    /// ドメインでの新規作成
    /// 不変条件を適用し、ドメインイベントを発行する
    pub fn new(year_month: YearMonth, amount: Price) -> (Self, Vec<DomainEvent>) {
        let id = Uuid::new_v4();
        let balance = Balance::from_price(&amount); // ← ドメインルール適用
        let budget = Self { id, year_month, amount, balance, set_at: Utc::now() };
        let events = vec![DomainEvent::BudgetSet { ... }];
        (budget, events)
    }

    /// DBからの再構築
    /// ルール適用なし・イベント発行なし・値をそのまま復元する
    pub fn reconstitute(
        id: Uuid,
        year_month: YearMonth,
        amount: Price,
        balance: Balance,      // ← DB に保存された残高をそのまま復元
        set_at: DateTime<Utc>, // ← DB に保存された日時をそのまま復元
    ) -> Self {
        Self { id, year_month, amount, balance, set_at }
    }
}
```

---

## 2つのコンストラクタの役割分担

| | `new()` | `reconstitute()` |
|---|---|---|
| **目的** | ドメイン上の新規作成 | DBからの復元 |
| **UUID** | 内部で生成 | 引数で受け取る |
| **Balance** | `amount` から計算（ドメインルール） | 引数で受け取る（保存済みの値） |
| **Utc::now()** | 内部で取得 | 引数で受け取る（保存済みの日時） |
| **ドメインイベント** | 発行する（`BudgetSet`） | 発行しない |
| **呼び出し元** | ユースケース | Infrastructure層のリポジトリ |

```rust
// ユースケース: new() を使う
let (budget, events) = Budget::new(year_month, amount);
budget_repo.save(&budget).await?;

// PostgresRepository（Phase 03）: reconstitute() を使う
let budget = Budget::reconstitute(
    row.id,
    YearMonth::new(row.year, row.month)?,
    Price::new(row.amount)?,
    Balance::new(row.balance),
    row.set_at,
);
```

`reconstitute()` は Infrastructure 層からしか呼ばれない。ドメイン層のコードが `reconstitute()` を呼ぶことはない。

---

## 最初の疑問への答え

「UUID・Balance・Utc::now() を外から渡した方が良いのでは？」という問いへの答えはこうだ。

- **Balance は渡してはいけない。** 「初期残高 = 予算金額」というドメインルールが外に漏れ、集約の不変条件が守れなくなる。
- **UUID と Utc::now() を外から渡したい場面は存在する。** それは「DBからの復元」という全く異なる目的であり、`reconstitute()` が担当する。

`new()` と `reconstitute()` を分けることで、「ドメイン上の作成」と「インフラ上の復元」という2つの全く異なる操作に、それぞれ適切な設計を当てられる。

Phase 03 で `PostgresBudgetRepository` を実装するとき、`reconstitute()` を追加してDBの行から Budget を組み立てる。そのとき「なぜ `new()` ではなく `reconstitute()` を使うのか」という問いへの答えは、この章で理解したことがそのまま使える。
