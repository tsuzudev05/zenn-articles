## はじめに

DDDで「バウンデッドコンテキスト」という言葉が出てくると、最初は抽象的でつかみにくい。

「コンテキストって何？境界ってどこで引くの？コンテキストマップって？」

ホシカ（欲しいものリスト × 予算管理アプリ）で実際にコンテキストを識別した過程と、コンテキスト間の「統合パターン」について、コード例を交えながらまとめる。


## バウンデッドコンテキストとは（おさらい）

**「言葉の意味が有効な境界」。**

同じ「アカウント」という言葉でも、営業部門では「顧客との取引関係」、システム部門では「ログイン情報」を指すことがある。一つのモデルで両方を表現しようとすると、どちらにとっても使いにくいモデルになる。

バウンデッドコンテキストは「この境界の中ではこの言葉はこういう意味」を明示する仕組みだ。


## ホシカの3コンテキスト

ホシカには責務の異なる3つの領域がある。

| コンテキスト | 責務 | 主なモデル |
|---|---|---|
| **WishList** | アイテムの登録・管理・優先順位 | `WishItem` |
| **ImpulsePrevention** | Inboxの管理・レビューによるステータス遷移 | `WishItem`（ステータス遷移） |
| **Budget** | 月次予算・残高・購入記録 | `Budget`, `PurchaseRecord` |


## 問題：コンテキストを分けたら、どう連携するか

コンテキストを分けると困ることがある。

「ImpulsePreventionはWishItemのステータスを変更したい。でもWishItemはWishListコンテキストのモデルだ。直接参照していいのか？」

「購入が完了したら（ItemPurchased）、BudgetにPurchaseRecordを記録したい。でもWishListとBudgetは別コンテキストだ。どうつなぐ？」

コンテキスト間でデータや操作を共有する方法が必要になる。それを整理したのが**統合パターン**だ。


## 統合パターンとは

コンテキスト間の「つなぎ方」を分類したもの。ホシカで使っている3つのパターンを説明する。

| パターン | 目的 |
|---|---|
| **Customer-Supplier（顧客-供給者）** | 上流が下流にモデルを提供する関係を明示する |
| **ACL（腐敗防止層）** | 下流が上流のモデルに引きずられないよう保護する |
| **Published Language（公開された言語）** | イベントという共通言語でコンテキスト間を疎結合につなぐ |


## パターン1：Customer-Supplier（顧客-供給者）

### 何をするパターンか

**上流（Supplier）が下流（Customer）にモデルを提供する関係を明示する。**

ホシカでは WishList が上流、ImpulsePrevention が下流だ。`WishItem` は WishList コンテキストが所有・定義しており、ImpulsePrevention はそれを参照する。

```
WishList [上流/Supplier]
    ↓  WishItemを提供する
ImpulsePrevention [下流/Customer]
```

### なぜ「明示」する必要があるのか

「参照するだけなら、ただimportすればいいのでは？」と思うかもしれない。

ただimportするだけだと、依存関係が暗黙的になる。WishListのモデルが変更されたとき、ImpulsePreventionへの影響が見えにくくなる。「このモデルを変えると、下流の誰かが壊れるかもしれない」という認識が薄れる。

Customer-Supplierパターンで関係を明示することで、**変更の影響範囲が設計レベルで可視化される**。


## パターン2：ACL（Anti-Corruption Layer：腐敗防止層）

### 何をするパターンか

**下流コンテキストが上流のモデルに直接依存しないよう、変換層を挟む。**

なぜ「腐敗防止」か。上流のモデルをそのまま使い続けると、上流の設計の都合が下流に染み込んでくる（腐敗する）。ACLはその汚染を防ぐ層だ。

### ホシカでの具体例

WishList コンテキストの `WishItem` は登録・管理の文脈で設計されている。

```rust
// WishList コンテキストの WishItem（上流のモデル）
pub struct WishItem {
    pub id: WishItemId,
    pub name: String,
    pub price: Price,
    pub status: WishItemStatus,
    pub category: Category,
    pub created_at: DateTime<Utc>,
    pub updated_at: DateTime<Utc>,
    // WishList文脈で必要な全フィールドがある
}
```

ImpulsePrevention がこれをそのまま使うと、`name` や `price` など「レビューに関係ないフィールド」まで意識することになる。上流が `created_at` を `registered_at` にリネームしたら下流も修正が必要になる。

ACLを挟むと、下流は「自分の文脈で必要なものだけ」に絞ったモデルを持てる。

```rust
// ImpulsePrevention コンテキスト側のACL
// impulse_prevention/acl/wish_item_translator.rs

/// ImpulsePrevention文脈での「レビュー対象アイテム」
/// WishListのWishItemから必要なフィールドだけを取り出した内部表現
pub struct ReviewableItem {
    pub id: WishItemId,
    pub status: WishItemStatus,
}

/// ACLの変換ロジック
/// WishList の WishItem → ImpulsePrevention の ReviewableItem
impl From<WishItem> for ReviewableItem {
    fn from(item: WishItem) -> Self {
        ReviewableItem {
            id: item.id,
            status: item.status,
            // name, price, category などは ImpulsePrevention には不要なので捨てる
        }
    }
}
```

```rust
// ImpulsePrevention のユースケース
// application/use_cases/review_wish_item.rs

pub struct ReviewWishItemUseCase {
    wish_item_repo: Arc<dyn WishItemRepository>,  // WishList側のリポジトリ
}

impl ReviewWishItemUseCase {
    pub fn execute(&self, id: WishItemId, decision: ReviewDecision) -> Result<(), DomainError> {
        // 上流からWishItemを取得
        let wish_item = self.wish_item_repo.find_by_id(id)?;

        // ACLで変換：WishItem → ReviewableItem（ImpulsePrevention文脈のモデル）
        let mut reviewable: ReviewableItem = wish_item.into();

        // ImpulsePrevention文脈でのロジックを実行
        reviewable.apply_review(decision)?;

        // 変更を上流に書き戻す（ステータス更新）
        self.wish_item_repo.update_status(reviewable.id, reviewable.status)?;

        Ok(())
    }
}
```

**ACLの効果：** 上流（WishList）のモデルがどう変わっても、`From<WishItem>` の変換部分だけを修正すればいい。ImpulsePreventionのドメインロジックには影響しない。


## パターン3：Published Language（公開された言語）

### 何をするパターンか

**イベントという「共通言語」でコンテキスト間を疎結合につなぐ。**

Customer-SupplierやACLは「同期的な参照」の話だった。Published Languageは「非同期・イベント駆動の統合」だ。

「WishItemが購入されたとき、Budgetに購入記録を残したい」という要件を考える。

**悪い実装：** WishList が Budget のサービスを直接呼ぶ

```rust
// ❌ これはやってはいけない
// WishList コンテキストが Budget コンテキストに直接依存している
pub fn purchase(id: WishItemId, price: Price) {
    self.wish_item_repo.update_status(id, WishItemStatus::Purchased);
    self.budget_service.record_purchase(id, price);  // ← Budget を直接呼んでいる
}
```

これだと WishList が Budget の存在を知ってしまう。「購入されたら、他に何かすべきことが増えた」たびに WishList のコードを変更しなければならない。コンテキストの境界が崩れる。

**Published Languageの実装：** イベントを発行するだけで、受信は各コンテキストが責任を持つ

```rust
// ✅ WishList コンテキスト：「何が起きたか」を発行するだけ
// domain/events/item_purchased.rs

/// ItemPurchasedイベント（Published Language）
/// これがコンテキスト間の「共通言語」になる
/// プリミティブ型（u64）を使い、ドメインオブジェクトを渡さないのがポイント
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ItemPurchasedEvent {
    pub wish_item_id: String,    // WishItemId をプリミティブに変換
    pub actual_price: u64,       // Price をプリミティブに変換
    pub purchased_at: String,    // DateTime をプリミティブに変換
}
```

```rust
// ✅ WishList コンテキスト：ユースケースでイベント発行
// application/use_cases/purchase_wish_item.rs

pub fn execute(&self, id: WishItemId, actual_price: Price) -> Result<(), DomainError> {
    let mut wish_item = self.wish_item_repo.find_by_id(id)?;
    let event = wish_item.purchase(actual_price)?;  // ← ドメインイベントが返る

    self.wish_item_repo.save(&wish_item)?;
    self.event_publisher.publish(event)?;  // ← 発行するだけ。Budgetを知らない

    Ok(())
}
```

```rust
// ✅ Budget コンテキスト：イベントを受け取って自分の責務を実行
// budget/application/handlers/item_purchased_handler.rs

pub struct ItemPurchasedHandler {
    budget_repo: Arc<dyn BudgetRepository>,
}

impl ItemPurchasedHandler {
    pub fn handle(&self, event: ItemPurchasedEvent) -> Result<(), DomainError> {
        // イベントのプリミティブをBudgetコンテキストのモデルに変換（ACL的な役割）
        let wish_item_id = WishItemId::from(event.wish_item_id);
        let actual_price = Price::new(event.actual_price)?;

        // Budget文脈でのロジック
        let mut budget = self.budget_repo.find_current_month()?;
        budget.record_purchase(wish_item_id, actual_price)?;
        self.budget_repo.save(&budget)?;

        Ok(())
    }
}
```

**Published Languageの効果：** WishListは「ItemPurchasedを発行した」だけで仕事を終える。将来「購入時にメール通知も送る」という要件が追加されても、WishListのコードは変更しなくていい。通知用のコンテキストが `ItemPurchasedEvent` を受信して処理するだけだ。


## 3パターンの使い分けまとめ

```
[WishList]                         [ImpulsePrevention]            [Budget]

WishItem を所有・定義
    │
    │  Customer-Supplier
    │  （WishListが上流）
    ↓
    WishItem をACLで変換
    ReviewableItem として扱う ─────────── レビュー実行
                                              │
                                              │ Published Language
                                              │ ItemPurchasedEvent を発行
                                              ↓
                                         イベントを受信
                                         PurchaseRecord を生成 ─────→ [Budget]
```

| パターン | 使う場面 | ホシカでの適用箇所 |
|---|---|---|
| **Customer-Supplier** | 上流・下流の関係を明示したいとき | WishList（上流）→ ImpulsePrevention（下流） |
| **ACL** | 上流モデルの変更影響を下流から遮断したいとき | ImpulsePrevention側でWishItemをReviewableItemに変換 |
| **Published Language** | コンテキスト間を疎結合にイベントでつなぎたいとき | ItemPurchasedイベントでImpulsePrevention → Budget をつなぐ |


## Shared Kernel（補足）

`WishItemStatus` は ImpulsePrevention コンテキストで意味を持つが、WishList コンテキストでも参照する。このように2つのコンテキストが同じモデルを共有する場合を **Shared Kernel** と呼ぶ。

```rust
// shared_kernel/wish_item_status.rs
// 両コンテキストから参照される共有モデル
#[derive(Debug, Clone, PartialEq)]
pub enum WishItemStatus {
    Inbox,
    NextToBuy,
    OnHold,
    Archived,
    Purchased,
}
```

完全に分離するとコンテキスト間でモデルのコピーが生まれ管理コストが上がる。MVPの段階では共有を許容し、将来コンテキストが独立デプロイ単位（マイクロサービス等）になる場合に分離を検討する判断をした。


## コンテキストを分けて何が変わるか

### 変更の影響範囲が明確になる

「予算計算ロジックを変える」ときに Budget コンテキストだけを見ればいい。`WishItem` のステータス遷移に影響しないことが保証される。

### 「このコードはどこに書くか」が決まる

コンテキストの責務が明確なので、新しいロジックを追加するときに迷いにくい。「これはどのコンテキストの責務か」という問いで判断できる。

### テストの境界が引きやすい

Budget コンテキストのテストでは `WishItem` の実装詳細を知る必要がない。`ItemPurchasedEvent` を直接作ってハンドラーに渡すだけでテストが書ける。

```rust
#[test]
fn 購入イベントを受け取ったらPurchaseRecordが作成される() {
    let event = ItemPurchasedEvent {
        wish_item_id: "item-1".to_string(),
        actual_price: 3000,
        purchased_at: "2026-06-13T00:00:00Z".to_string(),
    };

    let handler = ItemPurchasedHandler::new(InMemoryBudgetRepository::new());
    handler.handle(event).unwrap();

    // WishItem の実装を一切知らずにテストできる
}
```


## まとめ

統合パターンは「コンテキストを分けた後、どうつなぐか」を整理したものだ。

- **Customer-Supplier**: 上流・下流の関係を明示して変更の影響範囲を可視化する
- **ACL**: 変換層を挟んで下流が上流の都合に引きずられないよう保護する
- **Published Language**: イベントを共通言語にして、コンテキスト間を疎結合につなぐ

コードに落とすと「ACLは `From` トレイトで実装」「Published Language はイベント型をプリミティブで定義して発行・受信に分ける」という形になる。パターンの名前より、**「なぜその実装が必要か」という理由**を理解することが大事だ。
